Техническая документация: Локальная RAG-система

1. Концепция RAG и её обоснование

Базовые генеративные модели (LLM) не обладают знаниями о внутренней корпоративной документации. При обработке узкоспециализированных запросов они склонны к "галлюцинациям" — генерации правдоподобной, но фактически неверной информации.

Архитектура RAG (Retrieval-Augmented Generation) устраняет эту проблему путем изменения механизма формирования ответа. Система не использует внутреннюю память нейросети. Вместо этого алгоритм предварительно извлекает релевантный контекст из утвержденной базы документов, передает его в генеративную модель и жестко ограничивает рамки ответа предоставленным текстом.

2. Технологический стек

Для реализации проекта использованы следующие компоненты:

Фреймворк LangChain

Инструмент оркестрации пайплайнов. Управляет бизнес-логикой приложения: векторизацией запросов, взаимодействием с базой данных и формированием промптов (инструкций) для нейросети.

LLM Ollama + Saiga Llama 3

Генеративный модуль. Локальная среда исполнения, использующая русифицированную языковую модель. Отвечает исключительно за анализ предоставленного текста и синтез связного ответа на естественном языке.

Векторная БД ChromaDB

Подсистема хранения данных. Сохраняет документы в виде математических векторов (эмбеддингов). Обеспечивает возможность семантического поиска — нахождения релевантных фрагментов по смыслу, а не по прямому совпадению ключевых слов.

Cross-Encoder (Реранкер)

Модуль верификации. Дополнительная легковесная нейросеть, которая математически оценивает релевантность найденных фрагментов запросу пользователя. Предотвращает смешивание несвязанных фактов в итоговом ответе.

3. Архитектура проекта и структура директорий

Проект разделен на независимые функциональные модули для удобства сопровождения и масштабирования.

rag_project/
├── docs/ # Директория для загрузки исходных PDF-документов
├── chroma_db/ # Сгенерированная векторная база данных
├── prepare_data.py # Скрипт индексации: парсинг и векторизация документов
├── main.py # Консольное приложение: базовый интерфейс для отладки пайплайна
└── app.py # Web-интерфейс: production-версия приложения на базе Streamlit

Примечание к исполняемым файлам:
Скрипт main.py предназначен для разработчиков и инженеров. Он используется для тестирования качества поиска, оценки работы реранкера и вывода метрик в консоль.
Скрипт app.py предоставляет графический пользовательский интерфейс (GUI) с поддержкой выбора документов, кэшированием моделей и сохранением истории сессий для конечных пользователей.

4. Жизненный цикл обработки запроса

Инициация: Пользователь отправляет текстовый запрос через консоль или веб-интерфейс.
Векторизация (Embedding): Запрос преобразуется в числовой вектор с помощью мультиязычной модели.
Первичный поиск (Retrieval): ChromaDB извлекает из базы 15 фрагментов текста, имеющих наибольшее математическое сходство с запросом.
Переранжирование (Reranking): Модель Cross-Encoder анализирует первичную выдачу, оценивает логическую связь каждого фрагмента с вопросом и оставляет 3 наиболее релевантных абзаца.
Промптинг: Формируется системный контекст, включающий 3 лучших абзаца, исходный запрос пользователя и директиву, запрещающую использование сторонних знаний.
Генерация: Нейросеть (Saiga) обрабатывает промпт и синтезирует итоговый ответ.
Атрибуция: Пользователь получает сформированный текст с обязательным указанием первоисточников (название документа и номер страницы).
